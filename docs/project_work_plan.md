# Подробный план работы над проектом

## 1. Что строим

Проект строится вокруг прототипа банковского research assistant: системы, которая по заданной теме ищет открытые источники, отбирает полезную информацию, снижает информационный шум и формирует проверяемую аналитическую выжимку для бизнес-команды банка.

Базовый демо-сценарий из материалов проекта: исследование темы "Применение CLTV в иностранных банках".

Финальный результат проекта:

1. Рабочий прототип, где пользователь вводит тему исследования.
2. Система строит план исследования.
3. Система собирает кандидаты источников из открытых источников или из заранее подготовленного демо-набора.
4. Система чистит и фильтрует источники.
5. Система сохраняет evidence: источник, фрагмент текста, ссылку, дату, тип источника, релевантность и уровень доверия.
6. Система формирует аналитическую записку с цитируемыми утверждениями.
7. Система проверяет, что ключевые выводы подтверждены источниками.
8. Команда показывает прототип, архитектуру, метрики качества и ограничения банковского контура.

Главная идея проекта: не просто использовать LLM, FinBERT или Playwright, а построить банковски зрелый research-конвейер с безопасностью, проверяемостью, аудитом и заменяемыми моделями.

## 2. Исходные требования из проектных файлов

Из `docs/docs.pdf`:

- задача прикладная для крупного банка;
- нужен прототип автоматизированной аналитики;
- система должна искать открытые источники по теме;
- система должна отбирать полезную информацию;
- система должна убирать информационный шум;
- система должна формировать понятный результат для бизнес-команды;
- пример темы: применение CLTV в иностранных банках;
- итог может быть веб-сервисом, notebook или простым интерфейсом.

Из `docs/bank_research_assistant_solution.pdf`:

- решение нельзя строить как простой внешний чат-бот;
- внутренние данные банка не должны уходить наружу;
- наружу могут уходить только обезличенные поисковые запросы по публичной теме;
- каждый важный вывод должен иметь источник;
- LLM должна подключаться через gateway и быть заменяемой;
- аналитик остается в контуре принятия решения;
- приоритет у легитимных источников: API, RSS, официальные документы, отчеты, разрешенный web-доступ;
- MVP должен включать Research Planner, Source Collector, Noise Filter, Domain Classifier, Evidence Store, LLM Synthesizer и Quality Gate.

## 3. Целевая архитектура

### 3.1. Основные компоненты

1. UI / Notebook / API
   - принимает тему исследования;
   - принимает настройки: география, период, типы источников, глубина поиска, формат отчета;
   - показывает статус выполнения;
   - показывает итоговый отчет и таблицу evidence.

2. Sensitivity Classifier
   - проверяет пользовательский запрос;
   - блокирует или предупреждает, если запрос содержит клиентские данные, внутренние названия проектов, стратегические показатели или персональные данные;
   - для MVP может быть rule-based.

3. Research Planner
   - превращает тему в структуру исследования;
   - генерирует подзапросы;
   - задает ожидаемые блоки отчета.

4. Source Collector
   - собирает кандидаты источников;
   - для раннего MVP может работать с mock-набором URL;
   - для более зрелого MVP подключается к Search API, RSS и официальным сайтам.

5. Parser / Cleaner
   - скачивает HTML/PDF;
   - извлекает основной текст;
   - удаляет навигацию, рекламу, дубли, cookie-banner, служебный текст.

6. Noise Filter
   - отбрасывает SEO-страницы, рекламные страницы, дубли и нерелевантные материалы;
   - может совмещать эвристики, BM25, embeddings и простую классификацию.

7. Domain Classifier / Reranker
   - ранжирует материалы по релевантности банковской аналитике;
   - отличает реально полезный материал о CLTV в банках от общего текста про маркетинг.

8. Evidence Store
   - хранит источники и извлеченные фрагменты;
   - фиксирует ссылку, дату, тип источника, источник, score, версию отчета;
   - делает выводы проверяемыми.

9. LLM Gateway
   - единая точка подключения модели;
   - позволяет заменить модель: локальная Gemma, AlfaGen, GigaChat on-prem, OpenAI-compatible endpoint или другая внутренняя модель;
   - изолирует бизнес-логику от конкретного провайдера.

10. LLM Synthesizer
    - пишет аналитическую записку строго по evidence-фрагментам;
    - не должен свободно выдумывать факты;
    - формирует executive summary, кейсы, таблицу источников, unknowns и рекомендации для дальнейшей проверки.

11. Fact Checker / Critic
    - проверяет, что утверждения имеют источники;
    - ищет неподтвержденные цифры;
    - отмечает слабые выводы и противоречия.

### 3.2. Поток данных

1. Пользователь вводит тему: "Применение CLTV в иностранных банках".
2. Sensitivity Classifier проверяет, можно ли отправлять поисковые запросы наружу.
3. Research Planner строит план: определения, use cases, методы расчета, банковские кейсы, риски, данные, метрики, vendors.
4. Source Collector собирает 30-100 кандидатов источников.
5. Parser / Cleaner извлекает текст.
6. Noise Filter удаляет мусор и дубли.
7. Domain Classifier ранжирует источники по банковской релевантности.
8. Evidence Store сохраняет лучшие источники и фрагменты.
9. LLM Synthesizer пишет отчет на основе evidence.
10. Fact Checker проверяет отчет.
11. UI показывает отчет, evidence table, unknowns, confidence и экспорт.

## 4. Рекомендуемая стратегия реализации

Лучший путь: не начинать сразу с большого enterprise-продукта. Сначала нужно доказать ценность research-конвейера на одной теме, затем постепенно добавлять интерфейс, безопасность и банковские требования.

### Этап 0. Подготовка проекта

Цель: привести проект к форме, в которой команда может совместно работать и быстро показывать прогресс.

Что сделать:

1. Создать структуру репозитория:
   - `notebooks/` для экспериментов;
   - `src/` для кода прототипа;
   - `data/seed_sources/` для подготовленных источников;
   - `data/raw/` для скачанных страниц;
   - `data/clean/` для очищенного текста;
   - `reports/` для сгенерированных отчетов;
   - `docs/` для постановки, планов и материалов защиты;
   - `tests/` для проверок.

2. Добавить базовые файлы:
   - `README.md`;
   - `requirements.txt` или `pyproject.toml`;
   - `.gitignore`;
   - `docs/project_work_plan.md`;
   - `docs/demo_scenario_cltv.md`.

3. Зафиксировать демо-тему:
   - основная тема: CLTV в иностранных банках;
   - дополнительная тема для проверки универсальности: например, "Next Best Action в розничном банкинге".

Ожидаемый результат:

- репозиторий понятен внешнему проверяющему;
- ясно, как запустить прототип;
- ясно, какие файлы являются кодом, данными, отчетами и документацией.

### Этап 1. Notebook MVP

Цель: доказать, что конвейер работает end-to-end хотя бы в notebook.

Формат:

- один notebook или небольшой набор скриптов;
- тема фиксирована или вводится одной строкой;
- источники можно взять из подготовленного списка;
- отчет генерируется в Markdown.

Минимальные блоки:

1. Ввод темы.
2. Генерация research plan.
3. Сбор URL-кандидатов.
4. Извлечение текста.
5. Очистка текста.
6. Фильтрация и ранжирование.
7. Сохранение evidence table.
8. Генерация отчета.
9. Проверка отчета.

Почему это важно:

- notebook быстрее всего показать на защите;
- легче объяснить каждый шаг;
- проще отлаживать качество источников и фрагментов;
- не нужно сразу тратить время на полноценный frontend.

### Этап 2. Модульный Python-конвейер

Цель: перенести логику из notebook в переиспользуемые модули.

Предлагаемые модули:

- `src/config.py` - настройки проекта;
- `src/models.py` - структуры данных: Source, Chunk, EvidenceItem, ResearchPlan, Report;
- `src/sensitivity.py` - проверка чувствительных запросов;
- `src/planner.py` - Research Planner;
- `src/collector.py` - Source Collector;
- `src/parser.py` - HTML/PDF parser;
- `src/cleaner.py` - очистка текста;
- `src/filtering.py` - noise filter;
- `src/reranker.py` - domain reranker;
- `src/evidence_store.py` - evidence storage;
- `src/llm_gateway.py` - единый интерфейс к модели;
- `src/synthesizer.py` - генерация отчета;
- `src/quality_gate.py` - проверка отчета;
- `src/pipeline.py` - orchestration.

Почему так:

- каждый компонент можно тестировать отдельно;
- легче показать архитектуру;
- проще заменить mock-источники на Search API;
- проще заменить модель.

### Этап 3. Web/API MVP

Цель: сделать прототип удобным для демонстрации.

Вариант A: Streamlit

- быстрее всего сделать;
- удобно для школьного/студенческого прототипа;
- можно показать input, прогресс, таблицы и отчет;
- меньше времени на frontend.

Вариант B: FastAPI + простой frontend

- лучше показывает инженерный подход;
- можно сделать API endpoints;
- проще развивать до bank-ready prototype;
- больше времени на разработку.

Рекомендация:

- если времени мало, делать Streamlit;
- если есть время и команда разработчиков, делать FastAPI + минимальный frontend;
- backend все равно держать модульным, чтобы UI был только оболочкой.

## 5. Как реализовывать отдельные методы

### 5.1. Research Planner

Задача:

- превратить одну широкую тему в набор исследовательских направлений и поисковых запросов.

Пример для CLTV:

- что такое CLTV в банках;
- зачем банки считают CLTV;
- как рассчитывают CLTV;
- какие данные нужны;
- какие банковские кейсы существуют;
- какие риски есть у CLTV;
- какие метрики качества моделей CLTV используют;
- какие vendor-solutions есть на рынке;
- как CLTV связан с retention, cross-sell, CAC/LTV, next best action.

Реализация для MVP:

1. Rule-based шаблон:
   - заранее описать блоки исследования;
   - подставлять тему в поисковые запросы;
   - быстро и надежно для демо.

2. LLM-assisted planner:
   - модель получает тему;
   - возвращает JSON с блоками исследования и поисковыми запросами;
   - JSON валидируется через Pydantic.

3. Hybrid:
   - базовые блоки задаются шаблоном;
   - LLM добавляет уточнения;
   - команда получает управляемость и гибкость.

Рекомендация:

- начать с hybrid: шаблон дает стабильность, LLM добавляет адаптивность.

Технически:

- Python;
- Pydantic-модель `ResearchPlan`;
- prompt с требованием вернуть JSON;
- fallback на статический шаблон, если модель недоступна.

### 5.2. Sensitivity Classifier

Задача:

- не допустить отправку внутренних или клиентских данных во внешний поиск.

Что проверять:

- номера телефонов;
- email;
- ФИО;
- номера карт или счетов;
- ИНН/паспортные паттерны;
- слова вроде "внутренний проект", "стратегия банка", "клиентская база", "скоринг", "лимиты";
- слишком конкретные внутренние названия.

Реализация для MVP:

1. Regex:
   - телефоны;
   - email;
   - длинные числовые последовательности;
   - вероятные номера карт.

2. Keyword rules:
   - список чувствительных слов;
   - severity: low, medium, high.

3. Decision:
   - safe: разрешить;
   - warning: показать предупреждение;
   - blocked: запретить запуск.

Для bank-ready версии:

- подключить DLP-систему банка;
- добавить NER-модель для персональных данных;
- логировать срабатывания.

### 5.3. Source Collector

Задача:

- собрать 30-100 кандидатов источников по поисковым подзапросам.

Варианты реализации:

1. Mock source list
   - заранее подготовленный список URL;
   - лучший вариант для самого первого демо;
   - позволяет не зависеть от API-ключей и сетевых лимитов.

2. Search API
   - Yandex Search API, Tavily, SerpAPI или другой разрешенный API;
   - возвращает title, snippet, URL;
   - хорошо для воспроизводимого поиска.

3. RSS
   - подходит для новостей, отчетов и регулярных источников;
   - меньше юридических рисков.

4. Official source crawler
   - список разрешенных доменов: банки, консалтинговые компании, регуляторы, научные публикации;
   - можно собирать только с allowlist.

5. Playwright fallback
   - применять только если страница требует JS;
   - не делать ядром системы.

Рекомендация:

- MVP: mock list + опционально Search API;
- зрелая версия: Search API/RSS/allowlist + Playwright fallback.

### 5.4. Fetching

Задача:

- надежно загрузить страницу или PDF.

Инструменты:

- `httpx` для обычных HTTP-запросов;
- `curl_cffi` для страниц, где нужен browser-like fingerprint;
- `Playwright` для JS-рендеринга;
- `requests-cache` или своя cache-директория, чтобы не скачивать одно и то же повторно.

Правила:

- сначала пробовать простой HTTP;
- затем `curl_cffi`;
- затем Playwright;
- хранить статус загрузки;
- уважать robots.txt и условия доступа;
- не позиционировать anti-bot обход как ценность банковского продукта.

### 5.5. Parsing HTML

Задача:

- получить основной текст статьи или отчета.

Инструменты:

- BeautifulSoup для простого HTML;
- trafilatura для извлечения основного текста;
- readability-lxml как fallback;
- Jina Reader как внешний fallback только если он допустим по условиям проекта.

Подход:

1. Удалить script/style/nav/footer/header.
2. Извлечь title, main text, metadata.
3. Убрать короткие строки, меню, повторяющиеся блоки.
4. Сохранить raw и clean версии.

### 5.6. Parsing PDF

Задача:

- извлечь текст из PDF-отчетов, презентаций и research papers.

Инструменты:

- `pypdf` для простых PDF;
- `pdfplumber` для таблиц;
- OCR через Tesseract или другой OCR, если PDF состоит из изображений;
- отдельная пометка `ocr_required`.

Подход:

1. Попробовать извлечь текст через `pypdf`.
2. Если текста мало, пометить PDF как image-based.
3. Для image-based PDF запускать OCR или исключать из MVP.
4. Разбивать PDF на страницы и chunks.

### 5.7. Chunking

Задача:

- разбить длинные документы на фрагменты, которые можно искать, ранжировать и передавать LLM.

Реализация:

- chunk size: 700-1200 tokens или 3000-5000 символов;
- overlap: 100-200 tokens;
- сохранять `source_id`, `chunk_id`, `page`, `section`, `url`.

Почему важно:

- LLM не должна получать весь интернет-текст сразу;
- evidence должно ссылаться на конкретный фрагмент;
- reranking работает лучше на chunks, чем на огромных документах.

### 5.8. Noise Filter

Задача:

- удалить нерелевантные, рекламные, дублирующиеся и слабые источники.

Методы:

1. Heuristic rules
   - минимальная длина текста;
   - запрет страниц с большим количеством рекламных слов;
   - удаление страниц без фактического содержания;
   - фильтрация по языку;
   - фильтрация по дате.

2. Duplicate detection
   - нормализация URL;
   - сравнение title;
   - MinHash или simhash;
   - cosine similarity по embeddings.

3. BM25
   - классический текстовый поиск;
   - хорошо работает как быстрый baseline;
   - можно реализовать через `rank_bm25`.

4. Embeddings
   - считать embedding для query и chunk;
   - ранжировать по cosine similarity;
   - для MVP можно использовать sentence-transformers, если доступно.

5. Domain rules
   - повышать вес официальных банков, регуляторов, консалтинга, академических источников;
   - снижать вес SEO-блогов и рекламных landing pages.

Рекомендация:

- MVP: heuristics + BM25 + duplicate detection;
- улучшение: embeddings + domain classifier.

### 5.9. Domain Classifier / Reranker

Задача:

- отличить текст, реально полезный для банковской аналитики, от общего маркетингового текста.

Варианты:

1. FinBERT
   - можно использовать как финансовый сигнал;
   - ограничение: классический FinBERT часто англоязычный и больше про sentiment;
   - для CLTV в иностранных банках может быть полезен, но не как единственный метод.

2. Multilingual embeddings
   - подходят для английских и русских материалов;
   - можно ранжировать по близости к банковскому query.

3. RuBERT / multilingual classifier
   - полезно для русскоязычных источников;
   - можно дообучить на размеченных примерах, если есть время.

4. LLM reranker
   - модель оценивает источник по шкале 0-5;
   - дорого, но удобно для небольшого набора top-50.

Рекомендация:

- сначала rule-based + BM25;
- затем добавить embeddings;
- LLM reranker применять только к top-N, чтобы снизить стоимость.

### 5.10. Evidence Store

Задача:

- сохранить все, на чем строится отчет.

MVP-варианты:

1. SQLite
   - быстро;
   - не требует серверной БД;
   - удобно для demo.

2. JSONL
   - еще проще;
   - удобно смотреть вручную;
   - хуже для сложных запросов.

3. PostgreSQL + pgvector
   - лучше для зрелой версии;
   - хранит embeddings;
   - ближе к production.

4. Qdrant
   - хороший vector store;
   - полезен, если embeddings становятся центральной частью.

Рекомендация:

- для Notebook MVP: JSONL или SQLite;
- для Web/API MVP: SQLite;
- для bank-ready: PostgreSQL + pgvector или Qdrant.

Что хранить:

- `source_id`;
- `url`;
- `title`;
- `publisher`;
- `source_type`;
- `publication_date`;
- `retrieved_at`;
- `raw_text_path`;
- `clean_text`;
- `chunk_text`;
- `relevance_score`;
- `trust_score`;
- `query`;
- `pipeline_version`;
- `model_version`.

### 5.11. LLM Gateway

Задача:

- не привязывать систему к одной модели.

Интерфейс:

- `generate(prompt, system_prompt, temperature, max_tokens)`;
- `generate_json(prompt, schema)`;
- `summarize_evidence(evidence_items)`;
- `critique_report(report, evidence_items)`.

Варианты моделей:

- локальная Gemma;
- AlfaGen;
- GigaChat on-prem;
- OpenAI-compatible endpoint;
- mock LLM для тестов.

MVP:

- сделать mock gateway или один OpenAI-compatible клиент;
- все вызовы модели пропускать через один класс;
- не вызывать модель напрямую из разных частей кода.

Bank-ready:

- добавить allowlist моделей;
- логировать модель и параметры;
- отключить отправку внутреннего контекста наружу;
- добавить policy перед каждым внешним вызовом.

### 5.12. LLM Synthesizer

Задача:

- сформировать отчет строго по evidence.

Структура отчета:

1. Executive summary.
2. Что такое CLTV и зачем банкам.
3. Как считают CLTV.
4. Какие данные нужны.
5. Банковские use cases.
6. Риски и ограничения.
7. Таблица источников.
8. Unknowns.
9. Рекомендации для дальнейшей проверки.

Правила prompt:

- использовать только переданные evidence-фрагменты;
- не придумывать цифры;
- если информации нет, писать "не найдено в источниках";
- каждый ключевой тезис снабжать `source_id`;
- отделять факты от интерпретаций.

### 5.13. Quality Gate

Задача:

- проверить отчет перед показом пользователю.

Проверки:

1. Все ключевые утверждения имеют source_id.
2. В отчете нет неподтвержденных чисел.
3. Нет ссылок на источники, которых нет в evidence store.
4. Есть раздел unknowns.
5. Есть таблица источников.
6. Выводы не противоречат evidence.
7. Отчет не содержит чувствительных данных.

MVP-реализация:

- regex для поиска чисел без ссылок;
- проверка наличия source_id;
- LLM critic для семантической проверки;
- итоговый статус: pass/warn/fail.

## 6. Метрики качества

### 6.1. Precision@K

Что измеряет:

- долю действительно релевантных источников в top-K.

Как реализовать:

- взять top-10 источников;
- вручную разметить relevant/not relevant;
- посчитать долю relevant.

Цель MVP:

- Precision@10 >= 0.6 на демо-теме.

### 6.2. Citation Accuracy

Что измеряет:

- долю утверждений, корректно подтвержденных источниками.

Как реализовать:

- выделить 10-20 ключевых утверждений в отчете;
- проверить, есть ли source_id;
- вручную проверить, действительно ли источник подтверждает утверждение.

Цель MVP:

- Citation accuracy >= 0.8.

### 6.3. Hallucination Rate

Что измеряет:

- долю тезисов, которых нет в evidence.

Как реализовать:

- список ключевых тезисов;
- проверка на наличие подтверждения;
- manual review.

Цель MVP:

- hallucination rate <= 0.2.

### 6.4. Noise Reduction

Что измеряет:

- насколько хорошо система удаляет мусорные источники.

Как реализовать:

- посчитать количество источников до фильтрации;
- посчитать количество источников после фильтрации;
- вручную проверить, не удалены ли важные источники.

Цель MVP:

- удалить 30-60% слабых кандидатов без потери ключевых источников.

### 6.5. Time to Brief

Что измеряет:

- сколько времени занимает подготовка первой аналитической записки.

Цель MVP:

- до 5-10 минут для демо;
- при mock-источниках до 1-2 минут.

### 6.6. Human Acceptance Rate

Что измеряет:

- насколько отчет полезен аналитику без существенной переработки.

Как реализовать:

- дать отчет 2-3 людям;
- попросить оценить по шкале 1-5;
- зафиксировать комментарии.

## 9. Основные риски и как их закрывать

### Риск 1. Система превращается в обычный LLM-chat

Митигировать:

- строить отчет только по evidence;
- хранить source_id;
- показывать таблицу источников;
- иметь quality gate.

### Риск 2. Плохие или нерелевантные источники

Митигировать:

- seed-list надежных источников для MVP;
- BM25 и heuristics;
- domain reranker;
- ручная оценка Precision@K.

### Риск 3. Галлюцинации в отчете

Митигировать:

- строгий prompt;
- source_id для тезисов;
- LLM critic;
- проверка чисел без источников;
- unknowns вместо выдуманных фактов.

### Риск 4. Сложность внешнего поиска

Митигировать:

- начать с mock/seed источников;
- Search API подключать позже;
- для защиты честно объяснить, что collector заменяем.

### Риск 5. Банковская безопасность не раскрыта

Митигировать:

- добавить sensitivity classifier;
- описать data boundary;
- добавить audit log;
- объяснить LLM gateway и on-prem совместимость.

### Риск 6. Слишком сложный стек для команды

Митигировать:

- сначала notebook;
- затем модули;
- затем Streamlit;
- не начинать сразу с Kubernetes, SSO, DLP и production-БД.
